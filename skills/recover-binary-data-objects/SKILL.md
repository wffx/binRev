# Recover Binary Data Objects

## Purpose

Build the S03 data-object and code/data-boundary model that complements function recovery. This Skill owns the S03 gate for IDA-visible unidentified bit/data/data-island cleanup.

This Skill may directly request read-only IDA/IDA MCP access for data items, bytes, xrefs, literal pools, strings, and segment state. Do not ask for a separate human confirmation for read-only connection; do not write IDA state.

## Inputs

Require:

- `S01/region-map.json`
- `S02/ida-baseline-snapshot.json`
- `S02/address-space.json`
- accepted IDA checkpoint or IDA MCP session

## Workflow

1. Enumerate visible data and suspect data islands.
   - Strings and printable ranges.
   - Literal values referenced by code.
   - IDA data items, arrays, and unnamed globals when available.
   - Region-map ranges that IDA treats as data.
   - Gaps between functions or basic blocks.
   - Ranges after unconditional branch/return and before the next executable instruction.
   - IDA `DCD`/`DCQ`/undefined-byte ranges in code-bearing segments.

2. Detect object patterns.
   - Fixed-width arrays.
   - Pointer-looking tables.
   - Function-pointer candidates.
   - Descriptor-like constants and bitmasks.
   - Literal pools generated by ADR/LDR literal usage.
   - Runtime-address tables.
   - Instruction-encoding-like constants that should remain data because control flow skips them.
   - Alignment, padding, or embedded opaque blobs.

3. Perform full code/data boundary cleanup.
   - Inline data inside code-like regions.
   - Literal pools between functions.
   - Tables misidentified as instructions.
   - Code misidentified as data.
   - Values that decode as ARM64 instructions but are unreachable and referenced as data.
   - For the main `.text` body, use code-first recovery. Except for binary header/tail appendices or explicitly segmented non-text ranges, do not accept middle `.text` `DCQ`/`DCD`/`DCB` data islands as final cleanup.
   - For each suspect range, emit one of: `code`, `instruction_fallback`, `alignment_padding_at_boundary`, `external_tail_data`, or `blocking_unknown`. Legacy `literal_pool`, `pointer_table`, and `constant_table` classifications are allowed only outside the main text body or when a reviewed exception proves the target format really embeds data there.
   - If an executable segment also contains rodata/config/string/jump-table-looking values, do not rely on IDA's current data flags or data xrefs alone. First attempt 4-byte ARM64 instruction recovery; only non-text boundary/tail regions may remain data.
   - For structured-looking regions inside the `.text` middle, split qwords into 4-byte words. Decodable words become code; undecodable words become explicit `.inst`/`instruction_fallback` blockers, not qword data.
   - Repeated barrier/cache/atomic-looking decodes inside a data-dense region are valid code-first recovery evidence unless control-flow or file-boundary evidence proves the region is external data.
   - For every reported or sampled data-island address, record whether it is an item head or a tail. If it is a tail, record the owning item head/end and ensure the head has a reviewable name/comment. Treat head/tail ambiguity as a visibility defect even when IDA flags are no longer unknown.
   - Literal pools may begin at 4-byte alignment, so an address such as `base+4` can be the tail of an 8-byte `DCQ`; do not reinterpret that tail as unresolved code or separate data.

4. In forward-test labs, reconcile `.inst` fallback with a symbolized oracle only under `validation_only`.
   - Build the oracle relation from byte/function fingerprints and a documented address delta, not from symbol names alone.
   - Convert only target words that the oracle classifies as code and that survive target readback as valid 4-byte instruction candidates.
   - Re-run target word-state export after every apply pass; stop only when oracle-code candidates are zero or the remaining set is explicitly recorded as residual blockers.
   - Treat persistent mismatches as evidence of IDA item-head/tail confusion, stale fallback comments, function-boundary conflicts, or oracle/target segment mismatch. Do not silently mark them fixed.
   - Keep oracle names and symbol labels out of production S03 data artifacts; they belong only under `validation/oracle/` or a clearly marked forward-test report.
   - If the user/reviewer explicitly chooses to proceed despite a small residual set, emit `accepted-risk` records with exact addresses and downstream constraints. Do not remove the residuals from audit history.
   - The reusable production pattern is not oracle-specific: enumerate all middle `.text` data/bit/fallback words, attempt code-first conversion, re-export readback state, and keep iterating until the remaining set is either fixed, classified outside text-body scope, or accepted-risk.

5. Produce IDA data-definition proposal input.
   - Suggest data item, array, string, alignment, or undefine/redefine operations only as proposal records.
   - Include expected pre-state, target range, width/alignment, xrefs, and rollback notes.
   - Do not directly mutate IDA.

6. Preserve confidence.
   - Use `confirmed` only for structurally validated data.
   - Use `candidate` for plausible objects.
   - Use `blocking_unknown` where evidence is insufficient and S04 would be polluted by the unresolved code/data boundary.

## Outputs

Produce:

- `S03/data-objects.jsonl`
- `S03/data-islands.jsonl`
- `S03/code-data-boundary-audit.json`
- `S03/unresolved-regions.jsonl`
- `S03/records/recover-binary-data-objects.evidence.jsonl`
- `S03/records/recover-binary-data-objects.decisions.jsonl`
- `S03/records/recover-binary-data-objects.unknowns.jsonl`

## Boundaries

- Do not assign VM or platform structure names from business background.
- Do not force unknown bytes into code or data; if the boundary cannot be fixed, emit `blocking_unknown` and keep S03 from acceptance.
- Do not use external symbol or source knowledge.
- Do not directly modify IDA.
- Do not allow middle `.text` qword/data-island ranges to pass to S04 as harmless unresolved regions.
- Do not preserve middle `.text` literal pools merely because IDA created data xrefs; under this workflow they must be recovered as code or `.inst` fallback unless they are in a reviewed boundary/tail exception.
- Do not claim a user-reported bit/data issue is fixed unless the readback shows both structural flags and human-visible head/tail mapping.
